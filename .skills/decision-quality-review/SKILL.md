# Decision Quality Review

plan.md の設計判断が「支配軸の取り違え」に陥っていないかを検出する。

## 背景

Decision Quality (DQ) フレームワーク（SDG）は、意思決定の質を6要素で分解する。設計判断の失敗の多くは、**本来の支配軸 ≠ 採用した支配軸** という構造で説明できる。

## 9つの評価軸

| #   | 評価軸           | 定義                                             |
| --- | ---------------- | ------------------------------------------------ |
| 1   | 目的適合性       | 解くべき問題に対して解が適合しているか           |
| 2   | 制約適合性       | 機能・非機能要件を満たしているか                 |
| 3   | 実現可能性(技術) | 技術的に成立するか                               |
| 4   | 実現可能性(組織) | 組織のスキル・体制で運用・保守・進化させられるか |
| 5   | 品質影響         | 保守性・運用性・変更容易性・進化性などの -ility  |
| 6   | 時間効果(短期)   | 短期的な実装速度・即効性                         |
| 7   | 時間効果(長期)   | 長期的な技術的負債・Real Options                 |
| 8   | リスク・不確実性 | 不確実性とリスクの評価                           |
| 9   | 整合性           | 既存アーキテクチャ・規約・ADRとの整合            |
| 10  | 合意可能性       | ステークホルダーの合意と実行コミット             |

## 用語

- **支配軸**: いまの判断で最も重視すべき評価軸（原則1つ）
- **二次条件**: 支配軸ほど重視しないが最低水準を確保する軸
- **支配軸の取り違え**: 本来の支配軸 ≠ 採用した支配軸

## 変更の性質分類（目安）

| 変更の性質                      | 推定される支配軸   | 推定される二次条件  |
| ------------------------------- | ------------------ | ------------------- |
| 障害対応/hotfix                 | 時間効果(短期)     | 整合性              |
| 基幹刷新/大規模リファクタリング | 品質影響 or 整合性 | 実現可能性(組織)    |
| PoC/スパイク                    | 目的適合性         | 時間効果(短期)      |
| 新機能追加                      | 目的適合性         | 制約適合性 + 整合性 |
| セキュリティ対応                | リスク・不確実性   | 制約適合性          |
| 運用改善/DX向上                 | 品質影響(運用性)   | 実現可能性(組織)    |
| 技術的負債解消                  | 品質影響(保守性)   | 時間効果(長期)      |

この分類は目安であり、plan.md の文脈から判断して調整すること。

## レビュー手順

### Step 1: 変更の性質を分類する

plan.md の Goal / Background を読み、上記カテゴリから最も近い性質を選ぶ。あるべき支配軸と二次条件を特定する。

### Step 2: plan.md の実際の重点を読み取る

plan.md の Approach / 設計判断 を読み、実際にどの評価軸が強調されているか、どの軸が軽視されているかを特定する。

### Step 3: 支配軸のズレを検出する

Step 1 のあるべき支配軸と Step 2 の実際の重点を比較し、ズレがあれば上記の失敗パターンテーブルと照合する。

**候補パターンが見つかった場合**: `~/.claude/skills/decision-quality-review/references/patterns.md` を Read し、典型例と「敢えて選ぶとき」の条件を確認して、誤検出でないことを検証する。

## このレビュアーがやらないこと

- **個別変更の正当性検証** → scope-justification-reviewer の責務
- **個別主張の論理整合性チェック** → logic-validator の責務
- **アーキテクチャパターンの詳細検証** → architecture-strategist の責務
- **実装コードのレビュー** → 対象外（plan.md の設計判断のみ）

## 出力フォーマット

```markdown
## Decision Quality Review

### 変更の性質分類

- **分類**: <カテゴリ名>
- **推定される支配軸**: <軸名>
- **推定される二次条件**: <軸名>

### plan.md の実際の重点

- **強調されている軸**: <軸名と根拠>
- **軽視されている軸**: <軸名と根拠>

### 支配軸の整合性

- **判定**: aligned | misaligned
- **該当パターン**: <パターン名（もしあれば）>
- **偏りの方向**: <↑↓の説明>
- **補足**: <文脈に応じた補足、「敢えて選ぶとき」に該当する場合はその旨>

### verdict: pass | advisory
```

- `pass`: 支配軸のズレなし、または「敢えて選ぶとき」の条件に該当
- `advisory`: 支配軸のズレの可能性あり（参考情報として報告、ブロッカーにはしない）
